# 블로그 디자인, 나의 Taste를 묻다

기술 블로그를 만들고 싶었지만, 내가 원한 건 단순히 글을 올릴 수 있는 웹사이트가 아니었다. 
문서 사이트처럼 차갑지 않고, 그렇다고 감성만 남은 포트폴리오처럼 보이지도 않는 공간. 정보와 분위기가 함께 살아 있는, 조금은 에디토리얼한 블로그를 만들고 싶었다.

막상 시작해보니 어려운 건 구현보다도 방향이었다. 어떤 레퍼런스를 봐야 하는지, 어떤 도구로 색감과 무드를 잡아야 하는지, 어디서부터 “그럴듯한 시안”을 넘어서 “내 것 같은 디자인”이 되는지, 그리고 그 결과물을 어떻게 실제 서비스까지 연결할지 하나씩 부딪혀가며 배워야 했다.

이 글은 그 과정을 정리한 기록이다. 레퍼런스 사이트를 통해 취향을 구체화하고, Google Stitch로 기본 방향을 잡고, Claude cowork로 디자인을 다듬고, 마지막에는 GitHub Pages 배포 문제를 해결하기까지. 한 번에 완성된 결과보다, 그 사이의 판단과 시행착오를 중심으로 이야기해보려 한다.

## 레퍼런스를 보기 전까지는 내가 원하는 것도 정확히 몰랐다

처음부터 Figma나 코드 에디터를 열지 않았다. 먼저 한 일은 레퍼런스 사이트를 보는 일이었다. 이미 잘 만들어진 블로그, 에디토리얼 사이트, 브랜드 저널 형태의 웹사이트들을 훑어보면서 내가 어떤 화면에 끌리는지부터 확인했다.

이 과정에서 점점 기준이 생기기 시작했다.

- 기술 블로그지만 지나치게 기능적이지 않을 것
- 타이포그래피와 여백이 인상을 결정할 것
- 콘텐츠가 중심이되, 브랜드 감각도 함께 드러날 것
- “템플릿 같은 화면”보다는 “누군가의 취향이 들어간 사이트”처럼 보일 것

이 단계는 단순히 예쁜 사례를 모으는 일이 아니었다. 오히려 내가 좋아하는 것의 공통점을 말로 설명할 수 있게 만드는 과정에 가까웠다. 레퍼런스를 보다 보면 취향은 점점 선명해지고, 그 취향은 이후 디자인 결정을 내릴 때 기준점이 된다.

## Google Stitch는 정답을 주기보다 방향을 빨리 좁혀주는 도구였다

어느 정도 기준이 잡힌 다음에는 Google Stitch를 활용해 기본적인 색감과 분위기를 탐색했다. 여기서 기대했던 것은 완성형 디자인이 아니었다. 내가 원했던 건 빠른 탐색이었다. 여러 무드와 톤을 짧은 시간 안에 비교해보면서, 어떤 방향이 맞고 어떤 방향이 아닌지를 거르는 과정이 필요했다.

실제로 Google Stitch는 이 초기 탐색 단계에서 꽤 유용했다. 색감의 계열을 바꿔보거나 전체적인 분위기를 조정해보면서, 막연했던 감각을 좀 더 구체적으로 볼 수 있었다. “대충 이런 느낌”이 시각적인 초안으로 바뀌는 순간, 이후의 작업도 훨씬 빨라졌다.

다만 이 단계에서 분명해진 한계도 있었다. Stitch는 무드 탐색에는 좋았지만, 그것만으로는 내가 원하는 수준의 개성과 디테일까지 확보되지는 않았다. 결과물은 충분히 그럴듯했지만, 아직은 어딘가 “잘 만들어진 일반적인 시안”에 가까웠다. 내가 원했던 것은 그런 화면이 아니라, 내 블로그처럼 느껴지는 화면이었다.

## Claude cowork로 넘어가면서 ‘그럴듯한 시안’이 ‘내 블로그’가 되기 시작했다

그래서 다음 단계에서는 Claude cowork를 활용해 디자인을 한층 더 다듬었다. 여기서는 무드를 탐색하기보다, 구조와 디테일을 조정하는 일이 중심이 됐다. 타이포그래피가 주는 인상, 헤더와 푸터의 밀도, 글 목록과 글 본문의 위계, 여백의 리듬 같은 요소들을 반복해서 손봤다.

이 과정이 중요했던 이유는 명확했다. 초기 시안은 방향을 제안해줄 수 있지만, 결국 완성도를 만드는 것은 세부 조정이기 때문이다. 색상 하나, 텍스트의 위치 하나, 문단 간격 하나가 화면 전체의 인상을 바꾼다. Claude cowork를 통해 그런 미세한 차이들을 조정하면서부터 화면은 점점 “예쁜 샘플”이 아니라 “내가 운영할 사이트”처럼 보이기 시작했다.

돌아보면 이 과정에서 얻은 가장 큰 인사이트는 도구마다 잘하는 역할이 다르다는 점이었다. Google Stitch는 방향을 빠르게 넓고 가볍게 탐색하는 데 유리했고, Claude cowork는 그중 하나를 붙잡아 더 정교하고 개인적인 결과물로 밀어붙이는 데 유리했다. 한 도구로 모든 단계를 해결하려 하기보다, 단계에 따라 역할을 나눠 쓰는 편이 훨씬 현실적이었다.

## 디자인이 끝나자마자 바로 배포할 수 있을 줄 알았다

디자인과 구현이 어느 정도 마무리됐을 때, 이제 남은 일은 간단해 보였다. GitHub Pages에 올리면 끝이라고 생각했다. 정적 사이트 배포는 익숙한 선택지였고, 개인 블로그를 올리기에도 충분히 가벼운 플랫폼이라고 판단했다.

그런데 실제로는 여기서 예상보다 오래 막혔다.

처음에는 GitHub Pages가 정적 페이지만 서비스할 수 있으니, 지금 만든 사이트 구조와는 맞지 않는다고 생각했다. React로 만든 다중 페이지 구조였고, 라우팅도 있었기 때문이다. 겉으로 보기에는 “아, 이건 GitHub Pages로는 안 되겠구나”라는 결론이 꽤 자연스러워 보였다.

하지만 조금 더 들여다보니, 문제는 GitHub Pages 자체가 아니었다. 내가 만든 앱 구조와 GitHub Pages의 서빙 방식이 어긋나 있었던 것이 진짜 원인이었다.

## GitHub Pages에서 실제로 문제가 됐던 것들

### 1. `BrowserRouter`는 정적 배포 환경과 바로 맞물리지 않았다

React Router의 `BrowserRouter`를 그대로 쓰고 있었기 때문에 `/post`, `/about` 같은 경로에 직접 접근하거나 새로고침할 경우 GitHub Pages에서는 404가 발생할 수 있었다. 서버 라우팅을 처리해주는 애플리케이션 서버가 있는 것이 아니라, 정적 파일만 서빙하는 환경이기 때문이다.

즉, 앱은 SPA처럼 동작하고 있었지만 배포 환경은 그 전제를 이해하지 못했다.

### 2. `base` 경로를 제대로 설정하지 않으면 asset이 깨진다

GitHub Pages 프로젝트 사이트는 보통 루트 `/`가 아니라 `/blog/` 같은 하위 경로에서 서비스된다. 그런데 Vite 기본 설정 그대로 빌드하면 CSS와 JS가 `/assets/...`처럼 루트 기준으로 잡히면서 실제 배포 경로와 어긋난다.

그 결과는 단순하다. 페이지는 열리지만 스타일이나 스크립트가 정상적으로 로드되지 않는다. 디자인은 살아 있어도 실제 서비스는 깨져 보이게 된다.

### 3. 문제는 “불가능”이 아니라 “설정되지 않음”에 가까웠다

이 과정을 거치며 내 생각도 바뀌었다. 처음에는 GitHub Pages라는 플랫폼의 한계라고 봤지만, 실제로는 배포 환경에 필요한 설정을 아직 맞추지 않은 상태에 가까웠다. 이 차이는 중요하다. 플랫폼을 바꿔야 하는 문제와, 구조를 이해하고 맞추면 되는 문제는 대응 방식이 완전히 다르기 때문이다.

## 결국은 구조를 Pages에 맞추는 쪽으로 풀었다

최종적으로는 디자인을 유지하면서 GitHub Pages에 맞게 구조를 정리하는 방향으로 해결했다.

적용한 변경은 아래와 같았다.

- Vite production `base`를 `/blog/`로 설정
- Router에 `basename` 적용
- 빌드 시 `dist/404.html`을 함께 생성
- GitHub Actions 기반 Pages 배포 워크플로 추가
- 실제 배포 대상 저장소와 remote 경로 정리

이후에는 디자인을 거의 건드리지 않고도 GitHub Pages에서 블로그를 정상적으로 서비스할 수 있었다. 처음에 느꼈던 “이건 플랫폼이 안 받는다”는 인상과 달리, 실제 해결은 생각보다 구체적이고 기술적인 설정의 영역에 있었다.

## 이번 작업을 하며 남은 몇 가지 교훈

첫째, 디자인 작업과 배포 작업은 완전히 다른 성격의 일이다. 화면이 완성됐다고 해서 서비스 준비가 끝난 것은 아니다. 특히 정적 배포 환경에서는 화면보다 URL 구조, asset 경로, router 설정이 더 중요해지는 순간이 온다.

둘째, AI 도구는 하나의 만능 도구라기보다, 각 단계에서 역할이 다른 협업 도구에 가깝다. Google Stitch는 방향 탐색에, Claude cowork는 디테일 고도화에 더 적합했다. 같은 “AI 도구”라고 묶어서 보기보다, 어떤 단계에서 어떤 역할을 맡길 것인지가 더 중요했다.

셋째, 구현 중 막혔을 때 “이 플랫폼은 안 된다”라고 너무 빨리 결론내리지 않는 것이 중요했다. 실제로 많은 문제는 플랫폼의 한계보다도 현재 설정과 구조의 부조화에서 나온다. 그 차이를 구분할 수 있어야 더 빠르게 해결할 수 있다.

## 마치며

이번 블로그 작업은 단순히 개인 사이트 하나를 만든 경험이라기보다, 취향을 어떻게 구체화하고, AI 도구를 어떻게 역할별로 활용하고, 마지막에는 구현과 배포를 어떻게 연결해야 하는지를 배운 과정에 가까웠다.

레퍼런스 사이트를 통해 취향을 정리하고, Google Stitch로 초기 방향을 잡고, Claude cowork로 화면을 다듬고, GitHub Pages의 정적 배포 환경에 맞게 구조를 수정하는 흐름은 앞으로 다른 웹 프로젝트를 할 때도 충분히 재사용할 수 있을 것 같다.

결국 이번 경험에서 가장 크게 남은 것은 하나다. 좋은 결과물은 한 번에 완성되지 않는다. 방향을 잡는 단계, 다듬는 단계, 부딪히는 단계, 다시 정리하는 단계를 거치면서 비로소 자기 것이 된다.
